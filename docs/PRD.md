## 1. 제품 개요

### 1-1. 서비스 정의

**Moodlora**는 사용자가 현재 감정을 선택하면 같은 감정을 가진 사람들의 글을 모아 보여주는 **감정 기반 익명 소셜 플랫폼**이다.

### 1-2. 비전

> "감정을 숨기지 않아도 되는 공간. 같은 마음을 가진 누군가와 연결되는 경험."

기존 SNS가 '잘 보이기 위한 자기 과시'에 최적화되어 있다면, Moodlora는 **솔직한 감정 표현과 공감 연결**에 집중한다. 익명성을 보장하면서도 감정이라는 공통 분모로 사람들을 연결하고, 감정 변화를 추적할 수 있는 자기 이해 도구를 제공한다.

### 1-3. 목표 사용자

| 구분            | 설명                                                                                   |
| --------------- | -------------------------------------------------------------------------------------- |
| **주요 사용자** | 20~30대 직장인/대학생. 감정을 글로 표현하는 데 익숙하고, 실명 SNS 피로감을 느끼는 사람 |
| **보조 사용자** | 자기 감정을 이해하고 싶지만 전문 상담의 문턱이 높은 10~40대 전반                       |

#### 페르소나 A: 직장인 이지우 (28세, 마케터)

- **감정 패턴**: 월요일 불안, 업무 과부하 시 무기력, 주말 평온
- **사용 동기**: "오늘 너무 지치는데, 나만 이런 게 아니라는 걸 확인하고 싶다."
- **Pain Point**: 실명 SNS에서 부정적 감정 표현 시 지인 반응이 부담. 감정 일기는 쓰다 중단.

#### 페르소나 B: 대학원생 김민준 (24세, 석사)

- **감정 패턴**: 불안(논문/발표)과 무기력(실험 실패)이 주를 이룸
- **사용 동기**: "내 감정이 뭔지 정확히 모르겠다. 비슷한 상황의 글을 읽으면 정리가 될 것 같다."
- **Pain Point**: 감정 표현 자체가 어렵고, 익명이 아니면 절대 글을 올리지 않음

## 2. 문제 정의

### 2-1. 핵심 문제

| 문제                 | 설명                                                                      | Moodlora의 해결 방식                                            |
| -------------------- | ------------------------------------------------------------------------- | --------------------------------------------------------------- |
| **감정 표현의 장벽** | 실명 SNS에서는 부정적 감정(슬픔, 불안, 무기력)을 솔직하게 표현하기 어렵다 | 완전 익명 지원으로 심리적 안전감 제공                           |
| **공감 연결의 부재** | 같은 상황, 같은 감정을 가진 사람을 찾기 어렵다                            | 7가지 감정 필터 + 상황 태그로 동일 감정 보유자의 글을 즉시 탐색 |
| **감정 인식 부족**   | 자신의 감정 변화를 추적하고 돌아볼 수 있는 도구가 부족하다                | 감정 히스토리 차트로 패턴을 시각화                              |

### 2-2. 시장 기회

현대인의 정신 건강 문제는 증가 추세이나, 기존 플랫폼은 감정 공유에 적합하지 않다.

| 서비스       | 익명성             | 감정 기반 필터      | 감정 히스토리   | 공감 반응         |
| ------------ | ------------------ | ------------------- | --------------- | ----------------- |
| Instagram    | 없음               | 없음                | 없음            | 좋아요 1종        |
| 블라인드     | 직장 인증 필요     | 없음                | 없음            | 투표/댓글         |
| 에브리타임   | 학교 인증 필요     | 없음                | 없음            | 공감 1종          |
| **Moodlora** | **완전 익명 가능** | **7가지 감정 필터** | **차트 시각화** | **4종 맞춤 공감** |

### FR-01: 감정 온보딩 [P0]

> **사용자 스토리**: 처음 접속한 사용자로서, 현재 감정을 선택하면 그 감정에 맞는 피드를 바로 볼 수 있다.

| 수용 기준                                                                   |
| --------------------------------------------------------------------------- |
| `emotion_selected` 쿠키가 없으면 `/` 접속 시 `/welcome`으로 자동 리다이렉트 |
| 8가지 감정 카드가 감정명, 이모지, 색상과 함께 표시된다                      |
| 카드 선택 시 쿠키 저장 + Zustand 스토어 업데이트                            |
| 선택 후 `/feed`로 이동하며 해당 감정이 필터에 반영된다                      |
| Zustand persist로 새로고침 후에도 선택 감정이 유지된다                      |

---

### FR-02: 인증 시스템 [P0]

> **사용자 스토리**: 사용자로서, Google 또는 Kakao 계정으로 간편하게 로그인하거나, 로그인 없이 익명으로 피드를 탐색할 수 있다.

| 수용 기준                                                                                    |
| -------------------------------------------------------------------------------------------- |
| `/login` 화면에 Google / Kakao 로그인 버튼이 노출된다                                        |
| 로그인 완료 시 `auth.users` 생성 + `profiles` 테이블 자동 생성 (트리거)                      |
| `/auth/callback` 라우트가 OAuth 토큰을 처리하고 세션 쿠키를 설정한다                         |
| 비로그인 상태로 `/feed`, `/posts/[id]` 읽기 전용 접근이 가능하다                             |
| 비로그인 상태에서 공감/글 작성 시도 시 로그인 유도 토스트가 표시된다                         |
| 세션 만료 후 보호 경로(`/posts/new`, `/profile`, `/history`) 접근 시 `/login`으로 리다이렉트 |

---

### FR-03: 피드 (필터, 정렬, 무한 스크롤) [P0]

> **사용자 스토리**: 사용자로서, 감정과 상황 태그로 필터링하고 최신순/인기순으로 정렬된 게시글을 스크롤하며 탐색할 수 있다.

| 수용 기준                                                                   |
| --------------------------------------------------------------------------- |
| 감정 필터 변경 시 URL `searchParams`가 업데이트되고 피드가 재조회된다       |
| 상황 태그 필터는 다중 선택 가능하다                                         |
| 최신순(`created_at DESC`)과 인기순(반응 합산 DESC) 탭 전환이 동작한다       |
| 초기 로드 시 최신 게시글 20개가 표시된다                                    |
| 스크롤이 하단 200px 이내에 도달하면 다음 페이지가 자동 요청된다 (커서 기반) |
| 마지막 페이지 도달 시 "모든 글을 읽었어요" 메시지가 표시된다                |
| 로딩 중 스켈레톤 UI가 표시된다                                              |

---

### FR-04: 게시글 작성 (3단계 폼) [P0]

> **사용자 스토리**: 로그인한 사용자로서, 감정 -> 상황 태그 -> 본문의 3단계로 자연스럽게 감정을 글로 표현할 수 있다.

| 수용 기준                                                         |
| ----------------------------------------------------------------- |
| Step 1: 감정 선택 필수 (현재 감정이 기본값)                       |
| Step 2: 상황 태그 1개 이상 ~ 5개 이하 선택                        |
| Step 3: 본문 10자 이상 ~ 1,000자 이하, 실시간 글자 수 카운터 표시 |
| 익명 토글 활성화 시 작성자가 "익명"으로 표시된다                  |
| 제출 성공 시 작성된 게시글 상세 페이지(`/posts/[id]`)로 이동      |
| 제출 실패 시 에러 토스트 표시, 폼 데이터 유지                     |
| 게시글 작성 시 `emotion_logs`에 감정이 자동 기록된다              |

---

### FR-05: 게시글 상세 + 공감 반응 [P1]

> **사용자 스토리**: 사용자로서, 게시글을 읽고 4종 공감 반응(나도/이해해/힘내/공감)을 토글할 수 있다.

| 수용 기준                                              |
| ------------------------------------------------------ |
| 반응 버튼 탭 시 즉시 카운트 +1 (낙관적 업데이트)       |
| 이미 반응한 버튼 재탭 시 반응 취소, 카운트 -1 (토글)   |
| 4종 반응을 동시에 남길 수 있다 (같은 타입만 중복 불가) |
| 서버 응답 실패 시 낙관적 업데이트가 롤백된다           |
| `view_count`가 상세 조회 시 +1 증가한다                |

---

### FR-06: 댓글 및 대댓글 [P1]

> **사용자 스토리**: 로그인한 사용자로서, 게시글에 댓글을 남기고 다른 댓글에 대댓글을 달 수 있다.

| 수용 기준                                                       |
| --------------------------------------------------------------- |
| 댓글 1자 이상 ~ 500자 이하 제한                                 |
| 익명 옵션 선택 시 작성자가 "익명"으로 표시된다                  |
| 대댓글은 댓글 하위에 들여쓰기로 표시된다 (`parent_id` 자기참조) |
| 자신이 작성한 댓글은 삭제할 수 있다                             |

---

### FR-07: 감정 히스토리 [P1]

> **사용자 스토리**: 로그인한 사용자로서, 기간별 감정 변화를 차트로 확인하여 내 감정 패턴을 이해할 수 있다.

| 수용 기준                                                        |
| ---------------------------------------------------------------- |
| 7일 / 30일 / 90일 기간 필터가 동작한다                           |
| recharts AreaChart로 날짜별 감정 분포가 시각화된다               |
| 특정 날짜 포인트 선택 시 해당일 감정 로그 목록이 하단에 표시된다 |
| 히스토리 없음 시 안내 메시지와 감정 기록 유도 표시               |

---

### FR-08: 검색 [P1]

> **사용자 스토리**: 사용자로서, 키워드로 게시글을 검색하고 감정/태그 필터를 조합할 수 있다.

| 수용 기준                                 |
| ----------------------------------------- |
| 최소 2자 이상 입력해야 검색이 실행된다    |
| 감정 필터, 태그 필터와 조합 가능하다      |
| 검색 결과 없음 시 빈 상태 화면이 표시된다 |
| 검색 중 로딩 스피너가 표시된다            |

---

### FR-09: 추천 시스템 [P2]

> **사용자 스토리**: 사용자로서, 같은 감정의 인기 글과 보완 감정의 글을 추천받아 새로운 위로를 발견할 수 있다.

| 수용 기준                                                                 |
| ------------------------------------------------------------------------- |
| 피드 하단 "이런 글은 어때요?" 섹션에 추천 카드 최대 5개 표시              |
| 보완 감정 매핑: ANXIETY -> CALM, SADNESS -> JOY/EXCITEMENT, ANGER -> CALM, LONELINESS -> JOY/EXCITEMENT |
| 같은 감정 인기순 3개 + 보완 감정 2개 조합                                 |
| 이미 읽은 게시글은 추천에서 제외                                          |

---

### FR-10: 안전 기능 (신고, 숨기기, 차단) [P1]

> **사용자 스토리**: 사용자로서, 부적절한 게시글을 신고하거나 보고 싶지 않은 글/유저를 숨기거나 차단할 수 있다.

| 수용 기준                                                                                      |
| ---------------------------------------------------------------------------------------------- |
| 게시글 메뉴에서 신고 / 숨기기 / 작성자 차단 선택 가능                                          |
| 신고 시 사유 선택 필수 (SPAM / HARASSMENT / INAPPROPRIATE_CONTENT / FALSE_INFORMATION / OTHER) |
| 숨긴 게시글은 피드에서 즉시 제거 (`hidden_posts` 기록)                                         |
| 차단한 사용자의 게시글은 서버에서 필터링                                                       |
| 자신의 게시글에는 신고/숨기기/차단 메뉴 미표시                                                 |
| 동일 게시글 신고 5건 이상 시 자동 숨김 처리                                                    |

---

### FR-11: 프로필 [P1]

> **사용자 스토리**: 로그인한 사용자로서, 닉네임을 수정하고 내가 작성한 게시글 목록을 확인할 수 있다.

| 수용 기준                                                       |
| --------------------------------------------------------------- |
| 닉네임은 2~20자, 특수문자 제외로 수정 가능                      |
| 내 게시글 목록이 최신순으로 표시된다                            |
| 타인 프로필(`/profile/[userId]`) 방문 시 공개 게시글이 표시된다 |
| 익명으로 작성한 게시글은 프로필에서도 "익명" 처리               |

## 4. 비기능 요구사항

### 4-1. 성능

| 항목                     | 목표 수치            | 측정 방법                  |
| ------------------------ | -------------------- | -------------------------- |
| 피드 초기 로드 응답 시간 | 1,500ms 이하 (p95)   | Supabase Query 실행 시간   |
| 무한 스크롤 추가 로드    | 800ms 이하 (p95)     | TanStack Query devtools    |
| 페이지 최초 렌더링 (LCP) | 2,500ms 이하         | Core Web Vitals            |
| 동시 접속자 목표         | 500명 동시 세션 처리 | Supabase Free Tier 범위 내 |

### 4-2. 보안

| 항목                  | 정책                                                                                           |
| --------------------- | ---------------------------------------------------------------------------------------------- |
| RLS 적용              | 모든 테이블에 RLS 활성화. 기본: 비인증 접근 차단. 필요한 열람 권한만 명시적 허용               |
| 익명 신원 보호        | `author_id`는 DB에 저장하되, API 응답에서 익명 글의 경우 `author_id`를 null로 마스킹           |
| Service Role Key 격리 | `SUPABASE_SERVICE_ROLE_KEY`는 서버 사이드(Route Handler)에서만 사용. 클라이언트 번들 포함 금지 |
| 입력값 검증           | 모든 API 진입점에서 Zod 스키마 검증 후 Supabase 쿼리 실행                                      |
| 신고 처리             | 신고 접수 후 72시간 이내 검토 목표. 자동 숨김: 동일 게시글 5건 이상 신고                       |

### 4-3. 접근성

| 항목              | 기준                                                                |
| ----------------- | ------------------------------------------------------------------- |
| 최소 지원 뷰포트  | 375px (iPhone SE 기준)                                              |
| 반응형 전략       | 모바일 우선 (TailwindCSS). 768px 이상 데스크탑 레이아웃 보정        |

## 5. 기술 아키텍처 개요

### 5-1. 기술 스택

| 역할            | 기술                                        | 비고                                        |
| --------------- | ------------------------------------------- | ------------------------------------------- |
| 프레임워크      | Next.js 16.2.2 (App Router)                 | `params`/`searchParams` 반드시 `await` 필요 |
| UI              | React 19 + TailwindCSS v4 + shadcn/ui       | 모바일 우선 반응형                          |
| 데이터베이스    | Supabase PostgreSQL                         | RLS 기반 접근 제어                          |
| 인증            | Supabase Auth (`@supabase/ssr`)             | Google + Kakao OAuth                        |
| 서버 상태       | TanStack Query v5                           | 캐싱, 무한 스크롤, 낙관적 업데이트          |
| 클라이언트 상태 | Zustand (+ persist)                         | 현재 감정 전역 관리                         |
| 폼 관리         | react-hook-form + Zod + @hookform/resolvers | 3단계 게시글 폼                             |
| 차트            | Recharts                                    | 감정 히스토리 AreaChart                     |

### 5-2. 아키텍처 다이어그램

```
[브라우저]
  |
  |-- 클라이언트 컴포넌트 (React 19)
  |     |-- Zustand (감정 상태 + persist)
  |     |-- TanStack Query (서버 상태 캐싱)
  |     |-- react-hook-form (폼 관리)
  |
  |-- Next.js App Router (서버)
  |     |-- 서버 컴포넌트 (SSR)
  |     |-- Route Handlers (/api/*)
  |     |-- Middleware (세션 갱신 + 라우팅 보호)
  |
  |-- Supabase
        |-- PostgreSQL (RLS 활성화)
        |-- Auth (OAuth + 세션 관리)
        |-- Triggers (프로필 자동 생성)
```

### 5-3. 핵심 아키텍처 결정

| 결정                                  | 이유                                                                    |
| ------------------------------------- | ----------------------------------------------------------------------- |
| NextAuth 제거 -> Supabase Auth        | 인증과 DB를 하나의 서비스로 통합하여 1인 개발 복잡도 감소               |
| Prisma 제거 -> Supabase JS 클라이언트 | ORM 없이 자동 생성 TypeScript 타입으로 타입 안전성 확보. 배포 크기 감소 |
| 커서 기반 페이지네이션                | 오프셋 방식 대비 대량 데이터에서 일관된 성능 보장                       |
| 낙관적 업데이트 (공감 반응)           | 즉각적인 사용자 피드백으로 체감 속도 향상                               |
| Zustand persist (감정 상태)           | 새로고침/재방문 시에도 감정 선택 상태 유지                              |

## 6. 데이터 모델 요약

### 6-1. ER 관계

```
auth.users (Supabase 내장)
  |-- 1:1 -- profiles (트리거로 자동 생성)
                |-- 1:N -- posts
                |           |-- M:N -- tags (post_tags 조인 테이블)
                |           |-- 1:N -- reactions
                |           |-- 1:N -- comments (parent_id 자기참조)
                |
                |-- 1:N -- emotion_logs
                |-- 1:N -- reports
                |-- M:N -- hidden_posts
                |-- M:N -- blocks (blocker/blocked)
```

### 6-2. 테이블 상세

| 테이블         | 목적                   | 핵심 컬럼                                            | RLS 정책 요약                                                     |
| -------------- | ---------------------- | ---------------------------------------------------- | ----------------------------------------------------------------- |
| `profiles`     | auth.users 확장 프로필 | nickname, is_anonymous                               | 본인만 UPDATE. SELECT 전체 허용                                   |
| `posts`        | 게시글 본문 및 메타    | emotion, content, is_anonymous, view_count           | SELECT 전체. INSERT/DELETE 본인만                                 |
| `tags`         | 상황 태그 마스터       | name (UNIQUE)                                        | SELECT 전체. 변경 불가                                            |
| `post_tags`    | 게시글-태그 M:N        | post_id, tag_id                                      | posts 정책 연동                                                   |
| `reactions`    | 공감 반응              | post_id, user_id, type                               | INSERT/DELETE 본인만. SELECT 전체. UNIQUE(post_id, user_id, type) |
| `comments`     | 댓글/대댓글            | post_id, author_id, parent_id, content, is_anonymous | SELECT 전체. INSERT/DELETE 본인만                                 |
| `emotion_logs` | 감정 기록 히스토리     | user_id, emotion, note                               | 본인 레코드만 SELECT/INSERT                                       |
| `reports`      | 신고 내역              | reporter_id, post_id/comment_id, reason, resolved    | INSERT 로그인 사용자. SELECT 관리자만                             |
| `hidden_posts` | 게시글 숨기기          | user_id, post_id                                     | 본인 레코드만                                                     |
| `blocks`       | 유저 차단              | blocker_id, blocked_id                               | 본인 레코드만                                                     |

### 6-3. 커스텀 타입 (ENUM)

| 타입            | 값                                                                |
| --------------- | ----------------------------------------------------------------- |
| `emotion_type`  | JOY, EXCITEMENT, CALM, SADNESS, ANGER, LETHARGY, ANXIETY          |
| `reaction_type` | ME_TOO, UNDERSTOOD, CHEER_UP, EMPATHY                             |
| `report_reason` | SPAM, HARASSMENT, INAPPROPRIATE_CONTENT, FALSE_INFORMATION, OTHER |

### 6-4. 주요 인덱스

| 인덱스                  | 대상                                   | 용도                   |
| ----------------------- | -------------------------------------- | ---------------------- |
| `idx_posts_emotion`     | posts(emotion)                         | 감정 필터 쿼리 성능    |
| `idx_posts_created_at`  | posts(created_at DESC)                 | 최신순 정렬 성능       |
| `idx_comments_post_id`  | comments(post_id)                      | 게시글별 댓글 조회     |
| `idx_reactions_post_id` | reactions(post_id)                     | 게시글별 반응 집계     |
| `idx_emotion_logs_user` | emotion_logs(user_id, created_at DESC) | 사용자별 히스토리 조회 |

## 7. 구현 우선순위 / 로드맵

### 7-1. Phase 구분

1인 개발 환경을 고려하여 핵심 루프(감정 선택 -> 피드 -> 글 작성 -> 공감)를 우선 완성한 뒤, 부가 기능을 점진적으로 추가한다.

| Phase        | 목표               | 포함 기능                                                    | 우선순위 | 완료 기준                              |
| ------------ | ------------------ | ------------------------------------------------------------ | -------- | -------------------------------------- |
| **Phase 1**  | 기반 설정          | 패키지 설치, Supabase 연결, 상수/스토어, Providers, 레이아웃 | P0       | Supabase 연결 확인, 감정 상수 작동     |
| **Phase 2**  | 인증               | Google/Kakao OAuth, 콜백 핸들러, Middleware, useCurrentUser  | P0       | 소셜 로그인 -> profiles 자동 생성 확인 |
| **Phase 3**  | 온보딩             | EmotionPicker, 감정 선택 페이지, 쿠키 저장, 리다이렉트       | P0       | 감정 선택 후 `/feed` 이동 E2E 확인     |
| **Phase 4**  | 피드               | 피드 API, 감정/태그 필터, 정렬, 무한 스크롤, PostCard        | P0       | 필터 변경 + 스크롤 로드 동작 확인      |
| **Phase 5**  | 게시글 작성        | 3단계 폼, Zod 검증, 게시글 생성 API                          | P0       | 글 작성 -> 피드 노출 확인              |
| **Phase 6**  | 상세 + 공감 + 댓글 | 게시글 상세, 공감 반응 토글, 댓글/대댓글                     | P1       | 공감 토글 + 댓글 작성 E2E 확인         |
| **Phase 7**  | 감정 히스토리      | 감정 기록 API, 히스토리 API, AreaChart                       | P1       | 감정 기록 -> 차트 반영 확인            |
| **Phase 8**  | 검색               | 검색 API (Supabase ilike), 검색 페이지                       | P1       | 키워드 검색 결과 정확도 확인           |
| **Phase 9**  | 추천               | 같은 감정 + 보완 감정 추천 API, 추천 섹션 UI                 | P2       | 추천 카드 노출 확인                    |
| **Phase 10** | 안전 기능          | 신고/숨기기/차단 API, PostMenu, 다이얼로그                   | P1       | 차단 후 피드에서 해당 글 미노출 확인   |
| **Phase 11** | 프로필 + 마무리    | 프로필 페이지, 닉네임 수정, 스켈레톤/에러 바운더리 전체 적용 | P1       | QA 통과                                |

### 7-2. 릴리즈 마일스톤

## 8. 제약 사항 및 가정

### 8-1. 개발 제약 사항

| 항목                   | 내용                                                                                   |
| ---------------------- | -------------------------------------------------------------------------------------- |
| **1인 개발**           | 기획, 디자인, 개발, 배포, 운영을 1명이 담당. 복잡한 기능보다 핵심 루프의 완성도에 집중 |
| **Next.js 16.2.2**     | `params`/`searchParams`를 반드시 `await`해야 하는 Breaking Change 존재                 |
| **Supabase Free Tier** | 동시 접속 500명, 500MB DB, 1GB 스토리지 제한. 초과 시 Pro 플랜 전환 필요               |
| **OAuth 제공자 의존**  | Google/Kakao 서비스 장애 시 해당 로그인 불가 (상호 대체 가능)                          |

### 8-2. 기술적 가정

| 가정                                 | 근거                                                        |
| ------------------------------------ | ----------------------------------------------------------- |
| Supabase RLS로 서버 사이드 인가 충분 | Row Level Security가 API 계층의 권한 검사를 대체            |
| `ilike` 검색으로 MVP 충분            | Full-text search(pg_trgm 등)는 사용자 규모 증가 시 도입     |
| 감정 7가지로 MVP 충분                | 너무 세분화하면 선택 장벽 증가. 향후 사용자 피드백으로 확장 |
| 커서 기반 페이지네이션               | 게시글 수가 10만 건 이상으로 증가해도 일관된 성능 유지      |

### 8-3. 비즈니스 가정

| 가정                                                 | 비고                                         |
| ---------------------------------------------------- | -------------------------------------------- |
| 익명성이 감정 표현의 핵심 동인                       | 실명 의무 시 작성율 급감 예상                |
| 감정 히스토리가 재방문 동인                          | 일기/감정 추적 습관이 리텐션에 기여          |
| 4종 공감 반응이 단순 좋아요보다 유의미한 연결감 제공 | 감정별 세분화된 반응이 작성자에게 더 큰 위로 |

### 8-4. 외부 의존성

| 서비스                | 용도            | 장애 시 영향                    |
| --------------------- | --------------- | ------------------------------- |
| Supabase (PostgreSQL) | 전체 DB 및 RLS  | 서비스 전면 중단                |
| Supabase Auth         | OAuth 세션 관리 | 로그인/인증 불가                |
| Google OAuth          | 구글 로그인     | Google 로그인 불가 (Kakao 대체) |
